# DASPOWEAR — Объяснение проекта (для зачёта)

Этот документ написан для защиты проекта. Я постарался объяснить каждый кусок проекта простыми словами + дать готовые ответы на вопросы, которые скорее всего задаст преподаватель.

---

## 1 · Что это вообще такое

**Daspowear** — это работающий интернет-магазин с одним товаром (виниловая юбка-макси). Сайт состоит из двух частей:

| Часть | Что делает | Технология |
|---|---|---|
| **Frontend** | то, что видит пользователь в браузере | HTML + CSS + JavaScript (без фреймворков) |
| **Backend** | сервер: хранит товары, принимает заказы, общается с платёжной системой | Python + FastAPI |
| **База данных** | хранит товары и заказы | SQLite (один файл `daspowear.db`) |
| **Платежи** | приём денег | ЮKassa (sandbox / тестовый режим) |

Эти части общаются между собой по HTTP — фронтенд делает запросы (`fetch`) к бэкенду, бэкенд отвечает JSON-ом.

## 2 · Архитектура (что куда обращается)

```
┌───────────────┐   HTTP/JSON   ┌──────────────────┐    ┌───────────────┐
│   Браузер     │ ────────────► │   FastAPI        │ ──►│  SQLite DB    │
│  (frontend)   │ ◄──────────── │   (backend)      │ ◄──│  daspowear.db │
└───────────────┘               └──────────────────┘    └───────────────┘
                                        │ ▲
                                        ▼ │
                                ┌──────────────────┐
                                │   ЮKassa API     │
                                │   (sandbox)      │
                                └──────────────────┘
```

**Сценарий покупки по шагам:**

1. Пользователь открывает сайт → фронтенд делает `GET /api/products` → бэкенд достаёт товары из SQLite → возвращает JSON.
2. Пользователь выбирает размер → жмёт **Add to bag** → товар сохраняется в `localStorage` браузера (это «корзина»).
3. Жмёт **Checkout** → заполняет email + адрес → жмёт оплатить.
4. Фронтенд шлёт `POST /api/orders` с email + списком позиций.
5. Бэкенд:
   - **пересчитывает цену сам** (это важно! нельзя доверять цене с фронта — там её могут подделать через DevTools);
   - сохраняет заказ в SQLite со статусом `pending`;
   - идёт в ЮKassa и говорит «создай платёж на 15 000 ₽»;
   - получает от ЮKassa **ссылку на оплату** (`confirmation_url`);
   - возвращает эту ссылку фронту.
6. Фронт делает `window.location = confirmation_url` → браузер уходит на сайт ЮKassa.
7. Пользователь вводит данные тестовой карты (`1111 1111 1111 1026`) → ЮKassa списывает деньги (понарошку) → редиректит обратно на `success.html?order=<id>`.
8. Страница `success.html` спрашивает у бэкенда: «как там мой заказ?» → бэкенд **сам спрашивает у ЮKassa** статус платежа → видит `succeeded` → обновляет статус в БД → возвращает фронту → фронт показывает «Thank you».

> Шаг 8 решает важную проблему: что если webhook от ЮKassa не дошёл (мы локально, без публичного IP)? Мы всё равно узнаем результат, потому что сами идём за ним.

## 4 · Бэкенд — детальный разбор

### 4.1 `models.py` — таблицы базы данных

В SQL базе три таблицы:

- **`products`** — товары. Поля: `sku`, `name`, `price`, `images` (JSON-массив путей), `sizes` (JSON-массив: `["XS","S","M","L"]`), `materials`, `care` и т.д.
- **`orders`** — заказы. Поля: `order_uid` (наш внутренний ID типа `a1b2c3d4...`), `payment_id` (ID платежа от ЮKassa), `status` (`pending` / `succeeded` / `canceled`), email клиента, адрес, сумма.
- **`order_items`** — позиции в заказе (товар + размер + количество + цена). Связана с `orders` через внешний ключ.

Это классическая схема one-to-many: один заказ → много позиций.

Используется библиотека **SQLAlchemy** — это ORM (Object-Relational Mapper). Вместо `INSERT INTO products...` мы пишем `db.add(Product(name="...", price=15000))`. Python-объекты автоматически превращаются в строки таблицы.

### 4.2 `schemas.py` — валидация

Pydantic-схемы — это «контракт» данных. Когда приходит запрос `POST /api/orders`, FastAPI автоматически проверяет:
- есть ли поле `customer_email`?
- это вообще email?
- есть ли список `items`?
- в каждом item есть `product_id`?

Если что-то не так — клиент сразу получает понятную ошибку 422, и до бизнес-логики дело не доходит. Это **защита от мусорных запросов**.

### 4.3 `database.py` — подключение к БД

Два важных момента:
- `connect_args={"check_same_thread": False}` — нужен потому, что FastAPI многопоточный, а SQLite по умолчанию хочет один поток.
- `get_db()` — функция-зависимость (dependency) для FastAPI. В каждый эндпоинт автоматически передаётся новая сессия БД, которая закрывается в конце запроса. Это правильный паттерн — не держим коннект открытым между запросами.

### 4.4 `payments.py` — общение с ЮKassa

Тут всего две функции:

- **`create_payment(amount, ...)`** — идёт в ЮKassa и говорит «создай платёж». В ответ получает объект с `confirmation_url` — это ссылка, по которой пользователь должен пройти оплату. ЮKassa требует **идемпотентный ключ** (`uuid.uuid4()`) — это уникальный ID операции, чтобы случайные повторы запроса не создавали дубль-платёж.

- **`get_payment_status(payment_id)`** — спрашивает у ЮKassa: «как там платёж X?» Используется на странице успеха, чтобы синхронизироваться, если webhook не дошёл.

В `metadata` мы передаём наш `order_uid` — потом ЮKassa вернёт его в webhook'е, и мы поймём, какой именно заказ обновлять.

### 4.5 `main.py` — все эндпоинты

| Метод | URL | Что делает |
|---|---|---|
| `GET`  | `/api/products` | список всех товаров |
| `GET`  | `/api/products/{id}` | один товар |
| `POST` | `/api/products` | создать товар (для админки, без авторизации — это демо) |
| `POST` | `/api/orders` | создать заказ + инициировать платёж |
| `GET`  | `/api/orders/{order_uid}` | получить заказ + актуальный статус |
| `POST` | `/api/webhook/yookassa` | принимает уведомления от ЮKassa |
| `GET`  | `/` | health check |
| `GET`  | `/docs` | автоматическая Swagger-документация (это бонус FastAPI) |

**Ключевые места в коде, на которые стоит обратить внимание:**

1. **Перерасчёт цены на сервере** — в `create_order()` мы заново берём цену из БД, а не из запроса. Это защита от того, что кто-то откроет DevTools и попытается купить юбку за 1 рубль.

2. **CORS middleware** — браузер по умолчанию запрещает фронту с `localhost:5173` обращаться к API на `localhost:8000` (разные порты = разные «origin»). Поэтому в `main.py` подключён `CORSMiddleware`, который разрешает эти запросы.

3. **Lifespan-функция** — современный способ FastAPI запускать код на старте приложения. При первом запуске она проверяет, пустая ли БД, и если да — вызывает `seed()`, который добавляет юбку.

### 4.6 `seed.py` — первый товар

Это **единственное место**, где описан товар в данных. Если нужно добавить другую вещь — добавляешь словарь в `SEED_PRODUCTS` и запускаешь `python seed.py`. Скрипт умеет и **обновлять** существующие записи (если уже есть `sku` — перезаписывает поля), поэтому можно править цену/описание и просто перезапускать.

### 4.7 Webhook — на пальцах

Webhook — это «звонок наоборот». Обычно мы спрашиваем ЮKassa, а тут ЮKassa **сама звонит нам**: «эй, платёж по такому-то ID прошёл успешно!». Это надёжнее, чем периодически опрашивать.

Но webhook требует **публичный адрес** в интернете. Локально (`localhost:8000`) ЮKassa не может до нас достучаться. Поэтому либо:
- ставим ngrok (туннель локалки в интернет);
- или (как у нас сейчас) — обходимся опросом из `GET /api/orders/{id}`, который сам идёт в ЮKassa за статусом.

В коде webhook готов и работает — он просто не вызывается локально.

## 5 · Фронтенд — детальный разбор

Я **намеренно** не использовал React/Vue/Next.js. Для проекта с одной страницей товара это был бы оверкилл, а голый HTML+JS:
- мгновенно загружается (нет огромного JS-бандла);
- не требует `npm install`, сборщика, билда;
- легче читать и защищать на зачёте.

### 5.1 `js/api.js` — два объекта

- **`API`** — обёртки над `fetch()` для запросов к бэкенду. Все запросы централизованы здесь — если поменяется URL, правится в одном месте.
- **`Cart`** — корзина. Хранится в `localStorage` под ключом `daspowear_cart_v1`. Это значит, что **корзина переживает перезагрузку страницы**, но привязана к браузеру (на другом устройстве её не будет). Это нормально для простого магазина.
- **`fmt.rub(value)`** — форматирование цены под русскую локаль (`15 000 ₽` вместо `15000`).

### 5.2 `js/shell.js` — общая обвязка

Вместо того чтобы копипастить навигацию и футер в каждый HTML-файл, мы их рендерим из JS в специальные точки монтирования (`<div id="nav-mount"></div>`). Это упрощённый аналог компонентов.

Здесь же:
- **Drawer-корзина** (выезжает справа);
- **Checkout-overlay** (полноэкранный модал с формой);
- **Submit-логика чекаута** — собирает форму, шлёт `POST /api/orders`, делает redirect на ЮKassa.

### 5.3 HTML-страницы

- **`index.html`** — главная. Структура:
  1. **Hero** — слева крупный заголовок «A singular object of clothing», справа фотография.
  2. **Ticker** — бегущая строка с tagline'ами (стилистическая отсылка к Balenciaga).
  3. **Lookbook** — мозаика из 5 фото в editorial-сетке.
  4. **Shop** — карточки товаров (пока одна юбка). При наведении одна фотка плавно сменяется второй (`opacity`-переход) — это маленькая, но очень «модная» деталь.
  5. **Statement** — большая фраза в духе «We make one thing at a time, and we make it well».
  6. **Newsletter** — форма подписки (демо).

- **`product.html`** — слева вертикальная галерея из всех фото юбки (`object-fit: cover`), справа sticky-блок с названием, ценой, размерами, кнопками, описанием и аккордеоном (материалы / уход / доставка).

- **`success.html`** — страница после оплаты. Опрашивает бэкенд каждые 2.5 секунды, пока статус не станет финальным.

### 5.4 Дизайн — что я делал и почему

**Что просили:** минимализм, light, чистый студийный фон. **Референсы:** Balenciaga / Prada / Gucci. **Фото юбки:** студия, белый фон, чёрный глянцевый материал.

Что я заложил:

- **Чисто-белый фон** (`#ffffff`), без зернистости и кремовых оттенков. Совпадает с фоном фото — кадры «выливаются» в страницу.
- **Шрифты:**
  - `Fraunces` — современная вариативная антиква (variable serif), используется для заголовков. Дает «дорогой» оттенок без банальщины (Times New Roman, Playfair и подобное было бы скучно).
  - `Inter` — нейтральный sans-serif для всего остального. Это honest workhorse — то, что используют Stripe, Vercel и многие fashion-сайты.
- **Большой акцент на типографику**, маленькие eyebrow-метки UPPERCASE с межбуквенным интервалом — это узнаваемый язык luxury-сайтов.
- **Грид-сетка**: 12 колонок, фото лукбука занимают разные части (6+6, 6+3+3, full-width strip) — это создаёт editorial-ритм, как в журнале.
- **Минимум цветов**: чёрный + белый + серый. Только в ошибках появляется приглушённый красно-кирпичный (`#c63d12`).
- **Анимации только функциональные**: corner-cases (выезд drawer'а, hover на карточке, перебивка фоток). Никаких летающих частиц или fade-in'ов на каждый абзац — это противоречит «quiet luxury».

## 6 · ЮKassa — как это работает

ЮKassa — это **платёжный шлюз** (payment gateway). Они дают нам:
- библиотеку Python `yookassa` (мы поставили её через pip);
- два ключа: **shopId** (публичный, идентифицирует магазин) и **secret key** (секретный, доказывает что мы — это мы);
- два «контура»: **sandbox** (тестовые платежи, ненастоящие деньги) и **production** (боевой).

У нас стоит sandbox — поэтому ключ начинается с `test_`, а в кабинете все операции помечены «Тест».

**Полный жизненный цикл платежа:**

1. **Создание** — мы шлём `Payment.create({...})` с суммой, описанием и **объектом `receipt`** (см. ниже). ЮKassa в ответ: «вот тебе платёж со статусом `pending` и ссылка для оплаты».
2. **Оплата** — пользователь идёт по ссылке, вводит карту. ЮKassa проверяет карту, при успехе списывает деньги.
3. **Капчуринг** — мы попросили `"capture": true`, значит ЮKassa и спишет и сразу зачислит. Если бы было `false` — деньги сначала «холдились» бы (как при бронировании отеля), и нам нужно было бы их подтвердить отдельным запросом. Для одёжного магазина двухступенчатость не нужна.
4. **Фискальный чек** — поскольку мы передали `receipt`, ЮKassa автоматически пробивает чек через свою онлайн-кассу и отправляет его покупателю на email.
5. **Уведомление** — ЮKassa дёргает наш webhook со статусом `succeeded` или `canceled`.
6. **Параллельно** — наш `success.html` опрашивает наш бэкенд, который опрашивает ЮKassa. Так мы узнаём результат даже без webhook'а.

### 6.1 Фискальный чек (54-ФЗ) — почему он нужен

Федеральный закон №54-ФЗ обязывает любого продавца в России выдавать покупателю **электронный чек** при онлайн-оплате. За нарушение — штраф. Раньше для этого нужно было покупать онлайн-кассу (~30 000 ₽) и интегрироваться с оператором фискальных данных (ОФД).

ЮKassa упрощает это: они сами работают с ОФД, мы просто передаём состав заказа, и ЮKassa выпускает чек.

**Как это устроено в коде** (`backend/payments.py`):

```python
"receipt": {
    "customer": {
        "email": customer_email,   # сюда придёт чек
    },
    "items": [
        {
            "description": "Vinyl Maxi Skirt",
            "quantity": "1",
            "amount": {"value": "15000.00", "currency": "RUB"},
            "vat_code": 1,              # 1 = «Без НДС»
            "payment_mode": "full_prepayment",
            "payment_subject": "commodity",  # физический товар
        },
    ],
}
```

Поля чека:
- **`vat_code`** — код НДС. У нас `1` («Без НДС»), что нормально для самозанятого / ИП на УСН. В реальном бизнесе это согласуется с бухгалтерией.
- **`payment_mode`** — режим расчёта. `full_prepayment` = полная предоплата (мы получаем деньги до отправки товара).
- **`payment_subject`** — тип предмета расчёта. `commodity` = товар (для услуг было бы `service`).

**Защита формы:** в HTML email-поле помечено как `type="email" required`, плюс на бэкенде Pydantic-схема `OrderCreate` имеет тип `EmailStr` — невалидный email просто не пройдёт.

**Безопасность:**
- секретный ключ **только в `.env`**, не в коде, не в git;
- цена считается на сервере, фронт не может её менять;
- в `metadata` платежа мы передаём наш `order_uid` — это связь между «миром ЮKassa» и нашей базой.

## 7 · Деплой — как сайт оказывается в интернете

Локально сайт живёт по адресу `http://localhost:5173`, и его видишь только ты со своего мака. Чтобы другие люди могли заходить, нам нужно **разместить (задеплоить) код на сервере, у которого есть публичный IP**.

Мы используем **Render.com** — это **PaaS** (Platform-as-a-Service): даём им код, они сами поднимают Linux-сервер, ставят Python, запускают наш `uvicorn` и выдают URL. Раньше это нужно было делать руками (купить VPS → настроить nginx → certbot → systemd → ...), сейчас это 4 клика.

### 7.1 Архитектура продакшена

В отличие от локалки, где бэкенд и фронтенд работают на **разных портах** (8000 и 5173) и общаются через CORS, на проде у нас **один сервис** делает оба дела:

```
   Браузер (любой человек)
          │ HTTPS
          ▼
   https://daspowear.onrender.com
          │
          ▼
   FastAPI (uvicorn) — один процесс:
     • /api/*       → API-эндпоинты
     • /css/*, /js/*, /images/* → отдаёт статику фронтенда
     • /, /product.html, /success.html → отдаёт HTML
          │
          ▼
   SQLite файл (в контейнере)
          │
          ▼
   ЮKassa (внешний HTTP)
```

В `backend/main.py` это сделано через `app.mount("/css", StaticFiles(...))` и эндпоинт-функцию, которая отдаёт нужный HTML по URL. Получается: **один origin → нет CORS → нет конфигов → проще для прода**.

### 7.2 Переменные окружения (env vars)

В отличие от локалки, секретные ключи **не лежат в файле `.env`**. Render даёт интерфейс, где их вписываешь руками, и они инжектятся в процесс при старте. Так они не попадают в git и в логи. Это **best practice** — twelve-factor app (https://12factor.net).

Какие переменные нужны:
- `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY` — те же, что локально;
- `RETURN_URL` — куда ЮKassa возвращает после оплаты (на проде это `https://....onrender.com/success.html`);
- `ALLOWED_ORIGINS` — на всякий случай для CORS, если будешь дёргать API с других доменов.

### 7.3 Что говорить преподу про деплой

> «Развёрнуто на Render.com — PaaS-платформе. CI/CD простой: git push в main → Render автоматически пересобирает контейнер и перезапускает сервис. SSL-сертификат выдаётся автоматически через Let's Encrypt. Секреты лежат в Render-переменных, в репозитории `.env` исключён через `.gitignore`. Бэкенд и фронтенд обслуживаются одним FastAPI-процессом — статика монтируется через `StaticFiles`, что устраняет CORS и упрощает деплой.»

## 8 · Возможные вопросы преподавателя — и ответы

**В: Почему FastAPI, а не Django/Flask?**
О: FastAPI быстрый (асинхронный), даёт автоматическую валидацию через Pydantic и автоматическую документацию (Swagger на `/docs`). Для REST API это сейчас стандартный выбор.

**В: Почему SQLite, а не PostgreSQL?**
О: SQLite — встроенная база, один файл, никакой настройки. Для прототипа это идеально. Для продакшна перешёл бы на PostgreSQL, поменяв одну строку: `DATABASE_URL` в `.env`. SQLAlchemy абстрагирует от конкретной БД, код менять не пришлось бы.

**В: Зачем валидация на бэкенде, если она есть на фронте?**
О: Фронт можно обойти. Любой человек открывает DevTools и шлёт любой JSON напрямую. Сервер — последняя линия обороны, поэтому валидация (через Pydantic) и пересчёт цены (через SQLAlchemy-запрос) должны быть **только** на нём. Фронт-валидация — это UX, а не безопасность.

**В: Что такое CORS и зачем он тут?**
О: Cross-Origin Resource Sharing — браузерная политика, которая запрещает скриптам с одного origin (например, `localhost:5173`) делать запросы к другому (`localhost:8000`). Это защита от того, чтобы зловредный сайт читал данные с твоего банка. Чтобы наш фронт мог обращаться к бэку, в FastAPI подключён `CORSMiddleware` со списком разрешённых origins.

**В: Что такое idempotency key в ЮKassa?**
О: Уникальный ключ, который мы шлём с каждым запросом на создание платежа. Если из-за плохой связи запрос повторится — ЮKassa увидит тот же ключ и не создаст второй платёж, а вернёт тот же. Это защита от двойных списаний.

**В: Корзина — почему localStorage, а не в БД?**
О: Не требует авторизации пользователя. Достаточно для одной сессии. Если бы был аккаунт — корзину связали бы с user_id в БД (тогда она бы синхронизировалась между устройствами).

**В: Что такое webhook?**
О: HTTP-эндпоинт, который вызывает не пользователь, а другая система (в нашем случае — ЮKassa). Используется когда сервис хочет нас уведомить о событии без того, чтобы мы постоянно его опрашивали.

**В: Что бы ты добавил, если бы было больше времени?**
О: (1) Авторизация (логин/регистрация, JWT), (2) админ-панель для добавления товаров без правки кода, (3) валидация webhook-подписи ЮKassa, (4) PostgreSQL, (5) тесты на pytest, (6) свой домен и HTTPS-сертификат через Let's Encrypt.

**В: Email в форме чекаута — зачем он, кроме связи?**
О: Это email для **фискального чека** (54-ФЗ). По российскому законодательству при онлайн-оплате продавец обязан выдать кассовый чек. ЮKassa делает это за нас — мы в API-запросе передаём блок `receipt` с email клиента и списком позиций, после успешной оплаты ЮKassa сама формирует чек и отправляет его на этот email. См. `backend/payments.py`, функция `create_payment`.

**В: Куда задеплоен сайт? Почему именно Render?**
О: Сайт работает на Render.com — это PaaS (Platform as a Service), как Heroku. Бэкенд развёрнут как Web Service (Python + uvicorn), фронтенд — как Static Site. Связаны через переменные окружения `ALLOWED_ORIGINS` (CORS) и `DASPO_API_URL` в `config.js`. Render выбран потому, что у него есть бесплатный план и он умеет автоматический деплой из GitHub при push. Альтернативы: Railway, Fly.io, VPS у Timeweb/Selectel + nginx.

**В: Что такое 54-ФЗ?**
О: Федеральный закон «О применении контрольно-кассовой техники». Обязывает выдавать кассовый чек при любой розничной продаже, включая интернет-магазины. Чек должен содержать список товаров, цены, налоги. ЮKassa умеет генерировать такие чеки и отправлять покупателю — мы только передаём ей нужные данные в запросе на создание платежа.

## 9 · Как запустить

### Локально (на маке)
1. Распаковать архив.
2. Правый клик на `start.command` → «Открыть» (первый раз!).
3. Дождаться установки (~1 минута первый раз).
4. Браузер откроется на `http://localhost:5173`.
5. Тестовая карта ЮKassa: `1111 1111 1111 1026`, любой будущий срок, любой CVC.
6. Остановить: двойной клик на `stop.command`.

### Публично (доступно из интернета)
Полная инструкция в **`DEPLOY.md`** — там 6 шагов, занимает ~15 минут.
Кратко: код заливается на GitHub → Render.com подтягивает его → создаёт два сервиса
(бэкенд + статика) → выдаёт два публичных URL вида `https://daspowear.onrender.com`.
